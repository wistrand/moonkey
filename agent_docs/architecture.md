# Moonkey — architecture

Analog Connect IQ watchface for AMOLED Garmin devices (MARQ 2 / fenix 8). Monkey C.

## Layout
- `MoonkeyApp.mc` — `AppBase`; returns the view.
- `AnalogView.mc` — `WatchFace`; all drawing, layout, and config.
- `Astro.mc` (`module Astro`) — pure astronomy: moon (Meeus ch.47) + sun (Schlyter) ephemeris, bright-limb tilt, iterated moonrise/set. No device state — the view supplies location/time. Verified against `solunar`.
- `CalendarMath.mc` (`module Cal`) — pure Gregorian-date + DST math for the timezone field.
- `resources/` — strings, app settings (`settings/`), launcher icon, and the moon + animal source bitmaps (`drawables/`: moon, cat, fox, polar bear, seal).
- `moonkey.jungle` points sources at `src/`. `manifest.xml` lists the target devices and the Positioning permission.

## Render flow (`onUpdate`)
Single full redraw per tick. Order matters for layering: clear → moon + moonrise/set arc → day/night ring → cardinal data fields (incl. wind barb + precip bar) → diagonal radial fields → hands → hand cuts. The moon draws first because its bitmap's opaque corners would otherwise overlap the ring; everything else paints over them.

## Subsystems
- **Hands** — clipped to start outside the ring (no center spoke); each folds in the finer time units for smooth motion. **Differentiated by silhouette, not just length:** hour = tapered amber baton ending in an **open ring** (skeleton/pomme) tip — the longer minute sweeps through it; minute = tapered amber **lance** narrowing to a sharp point; second = white shaft with an amber tip (outer ~20%). A grey-under fill gives hour/minute a polished-bevel edge; the second hand uses a black outline. A thin black mark "cuts" hour/minute where they cross the outer ring, and the second hand at its colour split. `drawHourHand`/`drawMinuteHand` build the bodies from `handRect` (a tapered quad) + `lancePoly` + `drawCircle`; all `fillPolygon` points are integers. An optional **metal finish** (`metalHands`) draws the accent parts as **nested polys** (dark edges → bright centreline) for a cylindrical-metal gradient — solid when off or in AOD.
- **Day/night ring** — a 24-hour track with midnight at top, noon at bottom. An amber arc spans daylight (from computed sunrise/sunset); a pointer marks the current time, coloured by whether it's day or night. Surrounded by hour ticks and a gradient "sun ring" whose brightness peaks at solar noon. (Pointer/arc share the *same* angle map — `90 − 15·h` degrees — so they can't disagree; the pointer dot's ~5 px radius is ≈ ~24 min of arc on this small ring, so its edge can touch the sunset boundary while its centre is the true time.)
- **Data fields** — four cardinal readouts (N, S, weather, time+date) and four diagonal radial-text fields (NE, SE, SW timezone, NW), the latter drawn on a curve via a vector font (`drawRadialText`) and cached in a pool buffer (see Performance). **All seven data slots are configurable** (see Configuration): N, S, NE, SE, NW plus **W (weather)** and **E (time)** — each takes a complication or can be **hidden** ("none"); W/E default to their composite (weather widget / date+time). Defaults reproduce the original fixed layout (N heart rate, S steps, NE intensity, SE floors, NW next sun event). Two slots also have **special modes**: E can show a **Persian Solar date + Tehran clock**, N can show **custom text** (no value/heart/label). N/S draw the value with an adornment — a **heart icon** for `HEART_RATE` (above N / below S), else a tiny lowercase label (`cardinalLabel`); NE/SE/NW render inline `LABEL value` text (`compInline`, with sun types keeping the dynamic next-event `sunText`). The **weather** field stacks an icon over the temperature with a thin **precip-chance bar** between them, and a **wind barb** to its left:
  - *Weather glyphs* (`drawWeatherIcon`, all grey): clear (sun), cloud, rain (cloud+streaks), snow (cloud+dots), storm (cloud+lightning bolt), fog (stacked lines). `weatherCategory` buckets all ~54 `Weather.CONDITION_*` codes into those six; **"chance of" rain/snow codes map to `cloud`** on purpose so the precip bar — not the glyph — expresses the probability.
  - *Precip bar* (`drawPrecipBar`): a thin grey bar, fill ∝ `precipitationChance`, shown only when `>= 20%`.
  - *Wind barb* (`drawWindBarb`): a meteorological staff pointing **into** the wind (`windBearing` = direction FROM); feathers encode knots (half 5 / full 10 / pennant 50, capped 60 kt); skipped below 4 m/s (light/calm). The feathers **mirror to the opposite side of the staff in the southern hemisphere** (`observerSouthern`), per meteorological convention.
- **Moon** — phase and sky orientation, described below.

## Moon pipeline
The moon is the most involved element:
1. **Phase** — fraction through the synodic month from a known new-moon epoch.
2. **Shaded render (baked)** — the photo is drawn, then the unlit side darkened by a translucent-black overlay whose opacity ramps across the terminator for a soft edge; partial alpha keeps surface texture visible. The lit side is **left when waning, right when waxing** (a fixed ±90° baseline in the upright image). This per-scanline work is expensive, so it is baked into a cached `BufferedBitmap` and only re-rendered when the hour changes (the phase barely moves between bakes). The bitmap is re-created if the system reclaims it.
3. **Position & inclination** — the moon's geocentric RA/Dec come from **Meeus *Astronomical Algorithms* ch.47 principal terms** (`moonRaDec`, ~0.02°; an earlier Schlyter low-precision port ran ~6° off in declination, throwing moonset and tilt hours/degrees out). The terminator tilt as seen in the sky = **bright-limb position angle − parallactic angle** (`moonTilt`), which equals the true bright-limb angle measured CCW from the local vertical (verified against `/usr/bin/solunar` + an independent "lit side faces the Sun" computation). Computed in-code; baked with the shading **once per hour** (it drifts as the moon moves — hourly is plenty; a fresh GPS fix lands at the next bake).
4. **Composite** — the **rotation is baked into the buffer**, not done per frame: the upright shaded moon is rendered into a transient scratch buffer, then rotated into the persistent (opaque) display buffer via one `drawBitmap2` transform, point-sampled (not bilinear, to avoid bright-edge bleed). Because the image's lit side is a fixed ±90° baseline and `AffineTransform.rotate` is clockwise (screen y-down), the applied angle is **`tilt = (waxing ? −90° : +90°) − moonTilt()`** (so the lit side lands on the true PA); `MOON_TILT_OFFSET` is a 0° fine-tune. **Every frame is then just a plain `drawBitmap`** — no per-frame ephemeris or rotation (≈1 ms/frame; see Performance).
5. **Moonrise/moonset arc** (`drawMoonArc`/`moonRiseSet`) — a thin ring hugging the disc spanning the above-horizon span (moonrise→moonset), midnight-top like the day/night ring (colour is a setting, default light grey; a "Transparent" option hides it). The horizon hour-angle solve is **iterated** (`moonEqAt`/`horizonCross` Newton-refine the transit and re-evaluate the moon's RA/Dec at each crossing, since it moves ~0.5°/hr) off the same `moonRaDec` — ~7 Meeus evaluations, **hourly**, matching `solunar` to ~1 min (a single pass drifts >30 min near the horizon). Circumpolar / never-up guards; dropped in AOD. No now-pointer (the day/night arc already marks now).

**Numerical care:** the ephemeris is 32-bit-float sensitive. The Julian date subtracts the J2000 epoch (Unix seconds of 2000-01-01 12:00 UTC) as integers *before* dividing — adding a raw ~2.46-million Julian Date in `Float` quantizes to ~6-hour steps (tens of degrees of moon error). GMST is reduced mod-360 with whole revolutions dropped from the integer-day term so the large product stays precise.

Why baking: alpha-blended *fills* require `setStroke`/`setFill` (not `setColor`), a rotated terminator can't use axis-aligned scanlines, and palette buffers can't anti-alias — so the shading is baked upright in a full-colour scratch, the disc is rotated into the display buffer, and only the cheap blit happens per frame. The moon source photo has a slightly darkened limb so its bright rim doesn't survive on the shadowed side. The source bitmap is **selectable** (`moonImage`: moon / cat / fox / polar bear / seal, all 100×100); the phase shading and rotation apply regardless, so the animals also show "phases".

## Location & data sources
- **Moon inclination + moonrise/set** use `observerLoc()` = the observer's GPS (`Position`) first, weather observation point as fallback — so they track the real location and the sim's position control. Computed at the hourly bake (not per frame). `observerLoc` checks for an **out-of-range latitude** (`|lat| > ~90.5°`) — which only the simulator's uninitialised GPS produces (lat/lon = 180 after a fresh init; real GPS is always within ±90°). That case **substitutes a fixed default (Gothenburg, 57.7089/11.9746)** so the arc/tilt render in the sim without manually setting a position; the branch is dead code on a watch. If the location is genuinely null (no GPS *and* no weather), `moonTilt()` returns null → the bake draws the **upright phase** (terminator vertical) and the rise/set arc is dropped.
- **Sun times** (day/night arc + sunrise/sunset field) use `sunLocation()`, which prefers the **weather observation location** (GPS fallback). This is correct on-device, but in the **simulator** the arc's angle/width can diverge if the simulated weather location isn't your real location — the device is authoritative.
- Sun times and weather (condition, temperature, **`precipitationChance`**, **`windSpeed`/`windBearing`**) from the `Weather` API; the configurable field values from the **`Complications`** API (a change-callback caches them), with live `Activity`/`ActivityMonitor` fallbacks for HR/steps/intensity/floors. There is no moon/astronomy API, so phase, position, inclination, and rise/set are computed in-code (verified against `/usr/bin/solunar`).
- Local-hour mapping (day/night arc, SW timezone field) converts moments via `Gregorian.utcInfo` with an explicit offset; the day/night arc uses the device clock timezone, so a sim timezone that differs from the device shifts the arc position (not its width).

## Always-on (AOD)
The OS dims and pixel-shifts, so colours are kept. Only elements that add lit pixels without at-a-glance value are dropped when sleeping: the moon, second hand, hour ticks, sun ring, radial gradient arcs, hand cuts, filled heart, wind barb, and precip bar. Everything is gated on the sleep flag set by the enter/exit-sleep callbacks. (The hour open-ring and minute lance silhouettes persist in AOD, so the hands stay distinguishable even dimmed.)

## Performance
The whole face is re-rendered every second (the moving hands force it). Profiling (see [perf-analysis.md](perf-analysis.md)) found the per-frame hotspots *were* **radial vector text** (`drawRadialText`, ~7 ms/label) and **`drawArc`** — both now cached. Notes:
- **Radial labels + gradient arcs are cached in a graphics-pool buffer** — the biggest win. The four curved labels and the static gradient arcs render into one transparent dial-sized `createBufferedBitmap` (`_radBuf`, drawn once); each frame is a single `drawBitmap`. It re-renders only when a label string changes (~once/min) or on a sleep transition (the cache key holds both the field text and the sleep flag, so the awake-only arcs bake in or drop). This moves ~28 ms of `drawRadialText` + ~8 ms of gradient `drawArc` off the per-frame path.
- **The graphics pool is separate from the 128 KB heap** (CIQ 4, `enhancedGraphicSupport`): 3 MB on most targets, 4 MB on venu3. `createBufferedBitmap` allocates there — that's why the cache (and the moon buffers) fit. Pool buffers can be purged; every `ref.get()` is null-guarded and re-renders. A scratch probe confirmed `drawRadialText` renders into a non-palette pool buffer on AMOLED (the earlier "can't cache" / "alpha-buffer fails" notes were stale; only *palette* buffers can't anti-alias).
- **Constant geometry is precomputed once and reused** (no per-frame trig): the parametric heart curve, the 24-hour tick sin/cos, and the gradient-arc per-step angles+colours (`_gradArcs`). ~350 `sin`/`cos` calls/awake-second hoisted out.
- **Gradient arcs are real `drawArc`s** (into the cache buffer), angles/colours precomputed. Straight `drawLine` **chords** were tried as a cheaper substitute and **reverted** — faceting/seams looked bad.
- **The moon prebake** (above) cut its per-frame cost to ~1 ms.
- **Still live each frame:** the sun ring (`drawArc`, awake-only) — a candidate to fold into the cache later — and `clear` (~5 ms, unavoidable without partial updates).

## Configuration
All config is **Connect IQ app settings** (`resources/settings/{settings,properties}.xml`), editable in **Garmin Connect** on any device once installed as a beta/store app (and the simulator settings editor during dev). `AnalogView.applyProperties()` reads `Application.Properties`; `MoonkeyApp.onSettingsChanged()` re-applies on a Connect sync. It exposes:
- **Accent colour** (`accentColor`, curated list) — drives the hands/marks and the day/night arc.
- **Data colour** (`dataColor`, `_dataColor`) — the N/E/S/W readouts plus the weather glyph/barb/precip bar.
- **Moon-arc colour** (`moonArcColor`) — the above-horizon ring; a **Transparent** option (`-2`) hides it.
- **Moon image** (`moonImage`) — moon / cat / fox / polar bear / seal (the bake's shading + rotation still apply).
- **Custom text** (`text`) — a free-text string shown by the N field's "Show Text" mode.
- **Timezone** for the SW world clock (`tz`) — a curated zone picker (17 zones). `TZ_OFFSET`/`TZ_LABEL`/`TZ_DST` are indexed by the `tz` value. DST is computed per rule group (`dstHours`: EU/US/AU/NZ; no tz database, so fixed offset + rule), judged at ~02:00 local on the zone's standard-time date. **`tz == -2` hides the SW clock.**
- **Second-tick track** (`secTickColor`) — 48 short minor ticks + 12 longer U-marks at the 5-min positions, near the edge; a **None** option disables it (default light grey). Static, so baked into the radial cache (AOD-dropped).
- **Radial gradient** (`radialGradient`, default on) — toggles the four diagonal gradient arcs off; the flag is in the radial-cache key.
- **Small value font** (`smallValues`) — draws the non-radial values in the 0.8× `_labelFont` vector font via `vfont()` (an identity when off, so off is pixel-identical); the curved radial fields are unaffected.
- **Metal hands** (`metalHands`, default off) — gives the accent-coloured hands a cylindrical gradient via **nested polys** (dark edges → bright centreline; CIQ has no gradient fill); collapses to a solid fill when off or asleep.
- **Seven complication slots** (`compSE/NE/N/S/NW/W/E`, codes → `Complications.Id` via `compTypeFromCode`): SE height, NE energy, N/S body-or-movement, NW time/position, and **W (weather)** / **E (time)** as slots 5/6 that default to their composite (weather widget / date+time) or take a complication. Beyond plain types, several slots offer **sentinel modes** (codes >19, gated by flags rather than real complication types): **Persian Solar** (E or S, `100` — Jalali date + Tehran clock; small font both rows at S), **Show Text** (N, `101` — the `text` string), **Weather** (N or S, `102` — icon+temp on one line), **Date + Weekday** (E, `103`), **Steps + HR** (W, `104` — two icon-prefixed lines). `skipLabels` drops a W/E complication's dark-grey label and centres its value on the midline; when **both** W=Steps+HR and E=Date+Weekday, a compact mode shrinks all four data fields to the small font and aligns the E/W lines.
- **Sentinels:** colour/tz `-1` = unset (code default); complication `0` = unset/default, `>0` = a type, **`-1` = none/off** (`_compOff[slot]` hides the field). So every field is nullable from Garmin Connect.

**The native watch-face editor (`watchface-config.xml` + `AnalogWatchFaceDelegate`) was dropped.** Its complication picker can't represent the W/E *composite* default (it only offers "a type" or "cleared"), which made W/E unconfigurable-back-to-default and forced a `_compNativeOff` precedence layer. Garmin Connect settings already work on every device (including MARQ 2, which never had the editor), so the second surface — and the on-watch tap-to-config — were removed. ([finding-config.md](finding-config.md) has the original two-surface reasoning.)

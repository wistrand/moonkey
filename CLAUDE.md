# Moonkey — Connect IQ analog watchface

Garmin Connect IQ watchface ("Moonkey") in Monkey C. Targets AMOLED: `marq2aviator` (default), `fenix843mm`, `fenix847mm`, `marq2`, `venu3`, `epix2pro47mm`, `epix2` (epix Gen 2, on CIQ 5.2.0), `fr965`. `minApiLevel 4.2.1`. (vívoactive 5 was evaluated and dropped — no system vector fonts for the radial fields, and no `drawScaledBitmap`/native-format buffer for the moon bake.)

**Full design: [agent_docs/architecture.md](agent_docs/architecture.md).** Profiling & per-frame budget: [agent_docs/perf-analysis.md](agent_docs/perf-analysis.md). Configurability research (accent colour / timezone, sideload constraints): [agent_docs/finding-config.md](agent_docs/finding-config.md).

## Layout
- `manifest.xml`, `moonkey.jungle` at root. **Sources live in `src/`** (not the default `source/`); the jungle sets `base.sourcePath = src`.
- **Three app identities** — a published app and a beta app **cannot share an id** (per the dev portal: a beta is un-publishable, production must re-upload under a new id). Each has its own manifest + jungle + name (id, not name, is the device identity, so all three coexist on a watch):
  - **Production** `ec31f7e162fb48578ddc754d58d040bb` — `manifest.xml`, "Moonkey", the **permanent public Store id** (freeze it, never regenerate, always the same dev key). `make package` → `bin/moonkey.iq`. `make run`/`build`/`all` also use it (id is irrelevant in the sim).
  - **Beta** `a3f1c2d4e5b649a78c0d1e2f3a4b5c6d` — `manifest-beta.xml`, "Moonkey Beta" (`@Strings.AppNameBeta`), the release-candidate channel uploaded to the portal's "Beta Apps". `make package-beta` → `bin/moonkey-beta.iq`.
  - **Dev** `6ca23839dbb243279c8d7b4d3fe9b35b` — `manifest-dev.xml`, "Moonkey Dev" (`@Strings.AppNameDev`), local sideload. `make install` → `bin/moonkey-dev-<dev>.prg`.
  - Never upload `manifest-dev.xml`; never reuse the beta id for the public release.
- `src/MoonkeyApp.mc` — `AppBase`, returns the view.
- `src/AnalogView.mc` — all the drawing + config (`WatchFace`).
- `src/Astro.mc` — `module Astro`: pure moon/sun ephemeris (Meeus ch.47 + Schlyter sun), bright-limb tilt, iterated moonrise/set. The view passes location/time; verified against `solunar`.
- `src/CalendarMath.mc` — `module Cal`: pure Gregorian/DST date math (weekday/Sunday/doy/`dstHours`) + `toJalali` (Gregorian→Solar Hijri/Persian, arithmetic 33-yr cycle).
- `resources/` — strings + launcher icon + moon photo. `bin/` — built `.prg`s (one per device). `agent_docs/` — design + research notes.
- `setup-connectiq.sh` — one-time toolchain setup. `install.sh [device]` — sideload to a watch over MTP; `uninstall.sh [device|all]` — remove the sideloaded `.prg`(s) over MTP (the inverse; default removes all `moonkey-*.prg`). `screenshot.sh [--crop] [--transparent] [out.png]` — grab the sim window (`--crop` clips the menu header + status footer, applied before `--transparent`); `simrun.sh [device]` — build+run with settings overrides (env vars; used by `make run`); `auto-shot.sh [-t] [device] [out]` — build+run+wait-for-render+screenshot (`make shot`), and **honours the same env-var settings overrides as `simrun.sh`** (patches `properties.xml` + clears the `.SET` on restart, so e.g. `metalHands=true compW=104 make shot DEVICE=fenix843mm` shoots that config). `Makefile` — `make run`/`all`/`install`/`uninstall`/`package`/`shot`/`moon` (regenerates the moon bitmap from `data/moon-raw.jpg`).

## Build & run
SDK is managed by `~/go/bin/connect-iq-sdk-manager-cli` (open-source replacement for the GUI SDK Manager). Dev key: `~/.connectiq/developer_key.der`.

```bash
export PATH="$(~/go/bin/connect-iq-sdk-manager-cli sdk current-path --bin):$PATH"
monkeyc -d marq2aviator -f moonkey.jungle -o bin/moonkey-marq2aviator.prg -y ~/.connectiq/developer_key.der -w
connectiq &                                       # start simulator (once)
monkeydo bin/moonkey-marq2aviator.prg marq2aviator # load/reload app
```

- Simulator on Wayland: launch with `GDK_BACKEND=x11`. `monkeydo` cannot switch the device of a running sim — kill and relaunch the sim to change device.
- **Settings overrides for a run:** `make run` goes through `simrun.sh`, which honours env vars named after `properties.xml` `<property id>`s — e.g. `moonImage=1 make run` (cat), `tz=2 moonImage=2 make run` (fox + Stockholm). For each override it patches that default into `properties.xml` for the build, restarts the sim with the `.SET` cleared so the new default applies (the sim never overwrites an already-stored value), then restores `properties.xml`. No overrides ⇒ a plain build + load into the running sim.
- **Screenshots:** use `./screenshot.sh [--transparent] [out.png]`, not full-screen capture. GNOME/Wayland hides window geometry, but the sim is an XWayland client, so the script finds its X11 window by title (`xprop` → `WM_NAME` "CIQ Simulator") and grabs that window by id (`import -window`). Captures the window (not the screen) regardless of where it sits; crop the watch face *within* the result. No extra packages. **`--transparent`/`-t`** makes the white sim-window background transparent via a **seeded floodfill** (from confirmed-white edge points), so only white *connected to the background* is removed — the face's own white pixels (second hand, etc.) are enclosed by the black face and preserved. (A global `-transparent white` would wrongly erase those.)
- **Automated device shot:** `make shot DEVICE=<dev>` (or `./auto-shot.sh <dev> [out.png]`) builds, relaunches the sim, **waits for the face to actually render**, then screenshots. The sim emits no render-complete event, so it polls the window: the face is "rendered + settled" once a frame differs a lot from the blank baseline (>20% of pixels) **and** is then stable vs the previous frame (<2% — just the second hand). Frames are normalised to 240×240 before `compare` so size never matters. Typically ready ~8–9s after load. The final shot passes `--crop` (the render-detection baseline/frames stay full-window for stable diffing) so the menu header + status footer are clipped — the device area is found as the strongly-white-background row band.
- Devices must be downloaded **with fonts** (`device download -d <id> --include-fonts`) or the sim segfaults rendering text.
- Set HR in the sim via **Simulation → Health Monitoring**; weather/GPS via the Simulation menu (else sun/weather fields show `--`).

## Design notes
- **Always-on (AOD):** the OS dims + pixel-shifts, so colours are kept. Dropped when `_isSleeping`: moon + **moon arc**, second hand, hour ticks, sun ring, radial gradient arcs, hand cuts, filled heart, **wind barb, precip bar**; day-arc thinned. Gate every such element on `_isSleeping`.
- `ACCENT_COLOR = DAYLIGHT_COLOR` (amber). `RING_R_FRAC` drives both the day/night ring radius and the hand inner-clip.
- Day/night ring: midnight at top, noon at bottom; amber arc = daylight; pointer colour matches its side. (The current-time pointer is a filled dot whose ~5 px radius ≈ ~24 min of arc on this small ring — so its edge can visually touch sunset while its centre is the true time; not a bug.)
- **Hands clipped to start outside the ring** (no center segment); each folds in the finer time units for smooth motion; a black "cut" marks where hour/minute cross the outer ring. Differentiated by **silhouette**, not just length: hour = tapered amber baton ending in an **open ring** (skeleton/pomme) tip; minute = tapered amber **lance** to a sharp point; second = white shaft with an amber tip (outer ~20%). Grey-under-fill bevel on hour/minute; black outline on the second hand. `drawHourHand`/`drawMinuteHand`, with `handRect` (tapered quad) + `lancePoly` + `drawCircle`. **Metal-hand banding** (`metalHands` setting, default off → `_metalHands`): the accent fill is drawn as **nested polys** from full-width dark (`scaleColor(c, 0.45)`) to a thin bright centreline (`lerpColor(c, white, 0.55)`) — a cylindrical highlight (CIQ has no gradient fill). 5 bands on hour/minute, 3 each on the second hand's white shaft (grey→white) and amber tip; the hour open-ring stays solid. Collapses to a **single solid fill** when `_isSleeping` (AOD) or the toggle is off; no buffers, ~5 extra `fillPolygon`s/hand awake.
- **Weather glyphs** (`drawWeatherIcon`, all grey): clear (sun), cloud, rain (cloud+streaks), snow (cloud+dots), **storm (cloud+lightning bolt)**, **fog (stacked lines)**. `weatherCategory` buckets all ~54 `Weather.CONDITION_*` codes; **"chance of" rain/snow codes deliberately map to `cloud`** so the precip bar (not the glyph) carries the probability.
- **Wind barb** (`drawWindBarb`, left of the weather field): meteorological staff pointing **into** the wind (`windBearing` = direction FROM, compass deg); feathers = knots (half 5 / full 10 / pennant 50), capped 60 kt. Skipped entirely below **4 m/s** (light/calm). Feathers **mirror to the other side in the southern hemisphere** (`observerSouthern`, via `observerLoc`; cached after the first fix — must not leak a per-frame GPS lookup into the 1 Hz draw).
- **Precip bar** (`drawPrecipBar`): thin grey bar between glyph and temperature, fill ∝ `precipitationChance`, shown only when `>= 20%`.
- **Moon** (see architecture.md): soft-terminator shaded photo **with the sky-inclination rotation baked in**, into a cached display `BufferedBitmap` **once per hour**; every frame is a plain `drawBitmap` (~1 ms). Position is **Meeus ch.47 principal terms** (`moonRaDec`; the old Schlyter port ran ~6° off in declination — verify against `/usr/bin/solunar`). Inclination = bright-limb PA − parallactic angle (`moonTilt`). The baked image's lit side is fixed (**left waning / right waxing**, a ±90° baseline), so the bake rotates it onto the true angle: `tilt = (waxing?−90:+90) − moonTilt()` (`MOON_TILT_OFFSET` is a 0 fine-tune; `AffineTransform.rotate` is CW in screen y-down). **Julian date subtracts the J2000 epoch as integers first** — a raw ~2.46M JD quantizes to ~6 h in 32-bit float → tens-of-degrees error; GMST is likewise reduced mod-360.
- **Moonrise/moonset arc** (`drawMoonArc`/`moonRiseSet`): thin ring hugging the moon spanning the above-horizon span; midnight-top like the day/night ring; recomputed hourly; AOD-dropped. Colour is configurable (`moonArcColor` setting → `_moonArcColor`, default light grey); the **"Transparent"** option (`-2` → `_moonArcHidden`) drops the arc entirely. No now-pointer (the day/night arc already marks now). The solve is **iterated** — `moonEqAt`/`horizonCross` Newton-refine the transit and re-evaluate RA/Dec at each crossing (~7 Meeus evals, hourly), with circumpolar/never-up guards; **matches `solunar` to ~1 min** (a one-shot drifts >30 min near the horizon).
- **Configured via Connect IQ app settings only** (`resources/settings/` — `settings.xml` + `properties.xml`, editable in **Garmin Connect** for beta/store installs, or the sim's App Settings editor). Exposes **accent colour**, **data colour** (`_dataColor`, the N/E/S/W readouts + weather glyphs/barb/precip), **moon-arc colour** (`moonArcColor`, incl. **Transparent** = hide), **moon image** (`moonImage` → `_moonImage`: 0 moon / 1 cat / 2 fox / 3 polar bear / 4 seal, `Rez.Drawables.Moon`/`Cat`/`Fox`/`Polarbear`/`Seal`, all 100×100 — the bake's phase shading + rotation still apply, so the animals also show "phases"; changing it nulls `_moon` + forces a re-bake), a free-text **`text`** string (`_text`, shown by the N "Show Text" mode), **second-tick markers** (`secTickColor` → `_secTickColor`/`_secTickHidden`: a 60-position edge track — 48 short minor ticks (`_secMinor`) + 12 longer **U marks** (cap arc + 2 legs, opening **outward**) at the 5-min positions (`_secMajor`), `drawSecTicks`; **light grey default**, **None** `-2` = off — baked into the radial cache (`_radKey` includes the colour), AOD-dropped), a **radial-gradient toggle** (`radialGradient` → `_gradientOn`: off drops the gradient arcs from the radial cache; the flag is in `_radKey`), a **small-value-font toggle** (`smallValues` → `_smallValues`: draws all the *non-radial* values in the 0.8× `_labelFont` vector font via the `vfont(font)` helper — an identity when off, so off is pixel-identical; layout heights stay on the original fonts, only the glyphs shrink; vector cost per frame), a **timezone** for the SW world clock (`tz` → `TZ_OFFSET`/`TZ_LABEL`/`TZ_DST`, value == index), and **7 complication slots** (`compSE/NE/N/S/NW/W/E` → `compTypeFromCode`). `applyProperties()` reads `Application.Properties`; `MoonkeyApp.onSettingsChanged()` re-applies on sync.
  - **Slots & sentinels:** SE/NE/N/S/NW (data) + **W (weather)** and **E (time)** as slots 5/6 — W/E default to their **composite** (weather widget / date+time), or show a chosen complication. Complication code `0` = unset/default, `>0` = a type, **`-1` = none/off** (`_compOff[slot]` → field hidden). Colour/tz: `-1` = unset (code default); **`tz == -2` = hide the SW clock** (`_swOff`). Every field is thus nullable from Garmin Connect.
  - **The native watch-face editor was dropped** (`watchface-config.xml` + `AnalogWatchFaceDelegate` deleted): the editor's complication picker can't represent the W/E composite default (it only speaks "a type" or "cleared"), and a second config surface wasn't worth the precedence complexity. **All devices, incl. fenix 8, configure via Garmin Connect** (no on-watch tap-to-config).
  - **Sim gotcha:** `monkeydo`/`make run` sends only the `.prg`, so the simulator's **App Settings Editor stays empty** ("No settings file found") even though the build *does* emit a valid `bin/moonkey-<device>-settings.json`. Worse, that editor (`Sim_AppSettingsEditor.cpp`) is a **`wxWebView` that POSTs the descriptor to Garmin's `appSettings2/sdk/input` web service and renders the returned form** — so it needs BOTH the descriptor handed in by the **VS Code Monkey C extension** (build + Run; `monkeydo` doesn't push it) AND internet to `*.garmin.com`. There is no purely-local settings form. So: edit app settings in the sim only via the VS Code launcher (online); otherwise verify on a real **beta** install (same server pipeline) or, fully offline, change a `properties.xml` default and confirm `applyProperties()` applies it. **When changing a default, first delete the sim's persisted store** — `/tmp/com.garmin.connectiq/GARMIN/APPS/SETTINGS/MOONKEY-<DEV>.SET` — because the sim keeps the previously-seeded value and a new `properties.xml` default never overwrites an already-stored property (verified: stale `tz=-1` masked a new default of `2` until the `.SET` was removed). Not a bug — the settings are fine on-device.
- **Complication fields**: NE/SE/NW are inline `LABEL value` radial text (`compInline`, sun types keep the dynamic `sunText`); N/S draw the value with an adornment — **heart icon only for `HEART_RATE`** (above N / below S), else a tiny lowercase vector-font label (`cardinalLabel`, `_labelFont`). **W/E** are slots 5/6 (`drawDataFields`): their **composite** (weather widget / date+time) when `_compId == null`, else a `drawSideComp` label+value. The **`skipLabels`** boolean setting (`_skipLabels`) drops the dark-grey label on a W/E *complication* and lifts the lone value onto the watch's horizontal midline (`y = cy`, vs `cy+22` with a label). **E and S have a "Persian Solar" mode** (`compE`/`compS` sentinel `100` → `_ePersian`/`_sPersian`): the Jalali date for Tehran's date (`persianDateText`, e.g. "18 Khor", 4-letter month where possible) above the Tehran clock (`tehranTimeText`, UTC+3:30 no DST). At E the time is the big font; at S **both rows use the smaller font** (`drawPersianSmall`), shifted a line below centre. **N has a "Show Text" mode** (`compN` sentinel `101` → `_nText`): draws the `text` app-setting string (`_text`) centred, no value/heart/label. **N/S also have a "Weather" mode** (`compN`/`compS` sentinel `102` → `_compWeather[slot]`): the weather icon + temperature on **one line**, no wind barb, no label (`drawWeatherInline`). **E has a "Date + Weekday" mode** (`compE` `103` → `_eDateWeekday`): the date (as the default top line) over the 3-letter English weekday (`weekdayText`, via `Cal.weekday`). **W has a "Steps + HR" mode** (`compW` `104` → `_wStepsHr`, `drawStepsHr`): two small-font lines, persian-style padding (`fh*0.5`) — steps is a **centred value, no icon**; HR is a centred value with a small **heart** (`drawHeart`) just to its left. **When _both_ W=104 and E=103** a **compact** look kicks in: all four data fields (N/S/E/W) use the small font (`FONT_XTINY`, via `cardFont` in `drawCardinalComp`, incl. inline-weather temp) and the E/W lines align at `cy ± fh*0.5`. `slotValue` falls back to live HR/steps/intensity/floors; percentage types (battery/body-battery/pulse-ox) get a `%`. **`_compOff[slot]` hides a field entirely** (the `-1`/none setting); guards in `drawCardinalComp`/`compInline`/`nwFieldText` and the W/E arms.
- **Timezone DST** (`dstHours`, no tz database): EU (last-Sun-Mar→last-Sun-Oct), US (2nd-Sun-Mar→1st-Sun-Nov), AU/Sydney (1st-Sun-Oct→1st-Sun-Apr), NZ/Auckland (last-Sun-Sep→1st-Sun-Apr); southern zones wrap the year. Judged at **~02:00 local** on the zone's standard-time date (fractional day-of-year, so the changeover doesn't jump at midnight) via `weekday`/`nthSunday`/`lastSunday`/`doy`. UTC/Dubai/India/Tokyo/Moscow/Shanghai/Bangkok/Honolulu none; Tehran none (abolished 2022), São Paulo none (2019).

## Gotchas
- `dc.fillPolygon` only fills with **integer** points (floats render edges only); used for the heart curve and the hand bodies (`handRect` tapered quad, `lancePoly`).
- Radial text needs a `VectorFont` via `getVectorFont` (per-device faces; `RobotoCondensedBold` etc.); fields silently vanish if none match.
- HR for a watchface: `Activity.getActivityInfo().currentHeartRate` (live, on-device) with `ActivityMonitor.getHeartRateHistory` fallback (what the sim feeds).
- **Complication values are per-provider-scaled — don't trust them for counts.** The **STEPS** complication's `value` comes back **scaled to thousands as a Float** (`12.405` for 12405 steps), so formatting it gives "12". `slotValue` therefore reads the well-known activity types (HR/steps/intensity/floors) **straight from the device API** (`stepsText`/`heartRateText`/…, i.e. `ActivityMonitor`/`Activity` — the raw integer) and only uses the subscribed complication value for the other types. Relatedly, **never `format("%d")` directly on a `Float`/`Double`** — `.toNumber().format("%d")` (or `.format("%.0f")`) first, or large values mangle (same family as the temperature bug).
- `drawTicks` is currently unused (hour marks removed) — kept intentionally.
- **Alpha/blend:** the sim (and direct-to-Dc draws) ignore alpha via `setColor` and ignore blend modes; alpha-blended lines/fills need `setStroke`/`setFill` (32-bit AARRGGBB). A later `setColor` re-takes precedence (no clear method).
- Moon uses `Graphics.createBufferedBitmap` + `drawBitmap2` with an `AffineTransform` (point filter); observer location for inclination/rise-set is `Position` (GPS) first, weather fallback. Moon source has a darkened limb so its bright rim doesn't survive on the shadowed side.
- **Astronomy precision/verification:** the moon ephemeris is float-sensitive — keep the integer-first Julian date and mod-360 GMST (see Moon note). Cross-check any sun/moon change against `/usr/bin/solunar` plus a stdlib-Meeus recompute; `System.println` lands in `/tmp/monkeydo.log`. (Details in the `moonkey-astro-verification` memory.)
- **Observer location** for the moon (tilt + rise/set) and wind-barb hemisphere is `observerLoc()` = **GPS first, weather fallback**. An **out-of-range latitude** (`|lat| > ~90.5°`) only ever comes from the **simulator's uninitialised GPS** (lat/lon = 180 after a fresh/clear init) — real GPS is always within ±90°. So that case now **substitutes a fixed default location (Gothenburg, 57.7089/11.9746)** so the moon arc + tilt render in the sim without manually setting a position; the branch is dead code on a watch. If `observerLoc` is genuinely null (no GPS *and* no weather — not the 180 case), `moonTilt()` returns **null** and the bake draws the **upright phase** (terminator vertical, bright limb 90° CCW/left when waning), arc dropped.
- **Sun times use `sunLocation()` = weather observation location first** (GPS fallback) — unlike the moon (GPS-first via `observerLoc`). The device is authoritative; the day/night arc's angle/width can diverge **in the sim** if the simulated weather location isn't your real position. Local-hour mapping uses the device clock timezone, so a sim TZ ≠ device TZ shifts the arc position (not width).
- **Perf (see perf-analysis.md):** 128 KB heap cap, full redraw each second. `drawRadialText` (~7 ms/label) *was* the dominant per-frame cost; it is **now cached** (`drawRadialFields`/`renderRadialCache`): the four curved labels **and** the gradient arcs render into one transparent dial-sized `createBufferedBitmap` (`_radBuf`) and each frame is a single `drawBitmap`. Re-render only when a label string changes (~once/min) or on a sleep transition — the cache key (`_radKey`) includes both the field text and the sleep flag (arcs bake in when awake, drop when asleep). (`drawAngledText` straight-but-tilted text was a separate *visual* experiment, reverted — looked bad.)
- **Graphics pool, not the 128 KB heap:** `createBufferedBitmap` allocates from a separate **3 MB pool (4 MB venu3)** (`graphicsResourcePoolSize`, CIQ 4 `enhancedGraphicSupport`). The moon buffers and `_radBuf` live there. Pool buffers can be **purged** under pressure — always guard `ref.get()` for null and re-render (the moon and the radial cache both do). `drawRadialText`/`drawArc` render into a non-palette pool buffer fine on AMOLED (the old "can't cache" + "alpha-buffer fails" notes were overturned — see perf-analysis.md); palette buffers still can't anti-alias.
- **Constant geometry is precomputed once and reused every frame** (no per-frame trig): the parametric heart curve (`_heartHx/_heartHy`), the 24-hour ring tick sin/cos (`_tickSin/_tickCos`), and the gradient-arc per-step angles+colours (`_gradArcs`). The gradient arcs are drawn as **real `drawArc`s** into the cache buffer (straight-`drawLine` chords were tried for cost but **reverted — faceting/seams looked bad**). The sun ring still draws live with `drawArc`.
